배운 점 & 마무리

1. 제약이 품질을 만든다

AI 에이전트에게 자유를 주면 결과가 불안정해집니다. 이 프로젝트에서 가장 크게 배운 점은 잘 설계된 제약이 품질을 보장한다는 것입니다:

  • Rejection Loop — 7가지 규칙으로 빌드 전 기계적 검증
  • Quality Gate — 85점 미만은 배포 차단
  • CLAUDE.md — 선언적 행동 범위 정의
비유 고속도로에 가드레일이 있어야 안전하게 빠르게 달릴 수 있습니다. 가드레일 없는 AI는 탈선할 확률이 높습니다.

2. Dogfooding의 힘

자기가 만든 도구를 직접 쓰면 개선점이 명확하게 보입니다:

  • 이 테크톡 슬라이드를 Remarp로 만들면서 버그 3개 발견
  • VSCode Extension 프리뷰를 매일 쓰면서 UX 개선점 도출
  • "만들기만 하고 안 쓰는 도구"와 "매일 쓰는 도구"의 품질 차이는 극명

3. 지식을 코드로 축적

노하우가 문서(Confluence, Wiki)에만 머물면:

  • 찾기 어려움 — "어딘가에 있는데 어디였지?"
  • 실행이 번거로움 — "이 가이드의 명령어를 하나씩 복붙해야 해"
  • 업데이트 지연 — "이거 옛날 버전 문서 아니야?"

반면 Agent와 Skill로 캡슐화하면:

  • 한 줄 명령으로 즉시 실행
  • Git으로 버전 관리 → 항상 최신 상태
  • 팀 누구나 동일한 품질로 활용

4. 공유가 성장을 가속

혼자 쓰려고 만들었지만, 공유하니까 피드백으로 더 좋아졌습니다:

  • 동료의 "이 기능이 있으면 좋겠다"가 다음 버전의 로드맵
  • 다른 사람의 사용 패턴에서 예상치 못한 버그 발견
  • 오늘 이 테크톡도 그 공유 과정의 일부
한 줄 요약 "만들어 보고, 직접 쓰고, 공유하자" — 이 사이클이 도구와 만든 사람 모두를 성장시킵니다.
