# CallOps - AI-Assisted Call Center Architecture

> 콜센터 상담원을 실시간으로 보조하는 AI 시스템 아키텍처 구상

---

## 1. 문제 정의

### 현재 상황
- **상담원 숙련도 부족 + 실시간 응답 압박**: 신입 상담원 교육기간이 짧아 고객 질문에 즉각 응답이 어려움
- **방대한 비정형 지식**: Word, Wiki, PDF 등 사내 문서가 비정형으로 산재 — 전화 중 검색 불가
- **STT 인프라 구축 완료**: STT 엔진은 이미 운영 중이며 인식률 양호. 텍스트 스트림을 실시간으로 수신할 수 있는 상태
- **핵심 난관은 STT 이후**: 실시간 텍스트에서 **"지금 고객이 질문하고 있는가?"를 판별하는 것**이 기술적 핵심. STT 텍스트는 문장부호/억양 정보가 불완전하여 단순 키워드 매칭만으로는 부족. 한국어 종결어미의 문맥 의존성("~거든요"가 설명인지 질문인지)과 STT의 물음표 미출력 문제가 난이도를 높임
- **자체 CTI/PBX 환경**: Amazon Connect가 아닌 자체 CTI/PBX 사용 — STT 출력을 커스텀 인터페이스로 전달해야 함

### 해결해야 할 핵심 과제
1. STT 텍스트 스트림을 실시간으로 수집·분석하는 파이프라인 구축
2. 비정형 사내 문서를 검색 가능한 지식 베이스로 변환
3. 고객 질문에 대한 추천 답변을 2-3초 이내에 상담원에게 제시
4. 자체 CTI/PBX와의 원활한 연동
5. **실시간 응답 속도가 핵심**: 전화 통화 중 침묵 3초 이상은 고객 이탈/불만 → 질문 감지부터 추천 답변 표시까지 **2초 이내** 필수. 이 제약이 Option C(하이브리드) 선택의 핵심 이유

## 2. 핵심 아키텍처 흐름

```
고객 전화 → [자체 PBX/CTI] → [기존 STT] → 텍스트 스트림
    ↓
[WebSocket Gateway]                          ← ~50ms
    ↓
[실시간 처리 레이어] → 의도/질문 감지        ← ~100ms
    ↓
[RAG 파이프라인] → Hybrid Search + Rerank    ← ~300-500ms
    ↓
[Bedrock Claude] → 스트리밍 응답 생성        ← TTFT ~500ms
    ↓
[상담원 UI] → WebSocket push                 ← ~50ms
─────────────────────────────────────────
총 목표: 질문 감지 → 첫 토큰 표시 ≤ 1초
총 목표: 질문 감지 → 전체 답변 표시 ≤ 2초
```

### 데이터 흐름 요약
1. 고객 전화가 자체 PBX/CTI를 통해 수신
2. 기존 STT 엔진이 음성을 텍스트로 변환 (이미 구축 완료)
3. 텍스트 스트림이 WebSocket Gateway로 전달
4. 실시간 처리 레이어에서 고객의 의도/질문을 감지하고 대화를 요약
5. RAG 파이프라인이 관련 문서를 검색 (Hybrid Search)
6. Bedrock Claude가 검색 결과를 기반으로 스트리밍 응답 생성
7. 상담원 UI에 추천 답변, 출처 문서, 대화 요약이 실시간 표시

## 3. 자체 CTI/PBX 연동 설계

### 프로토콜 연동
- **STT → WebSocket/gRPC → Gateway Service (EKS)**
- 프로토콜 어댑터 패턴: CTI vendor별 플러그인 구조
  - 각 vendor의 STT 출력 포맷을 표준 메시지 포맷으로 변환
  - 새로운 CTI vendor 추가 시 어댑터만 구현

### 세션 관리
- **통화ID 기반 DynamoDB 세션 테이블**
  - Partition Key: `callId`
  - Sort Key: `timestamp`
  - 통화 메타데이터, 대화 이력, 상담원 ID 저장
  - TTL 설정으로 자동 만료 (24시간)

### 연결 생명주기
- **Heartbeat**: 통화 시작/종료 이벤트 수신
  - `CALL_START` → 세션 생성, WebSocket 연결 수립
  - `CALL_END` → 세션 종료, 대화 이력 아카이빙
  - `HEARTBEAT` → 연결 상태 확인 (30초 간격)

### 참고
- Amazon Connect 없이도 전체 파이프라인 동작 가능
- 향후 Amazon Connect 마이그레이션 시 Contact Lens 네이티브 연동으로 전환 가능

## 4. AWS 서비스 매핑 (3가지 옵션)

### Option A: Bedrock Knowledge Bases (Managed RAG)

| 구성요소 | AWS 서비스 |
|---------|-----------|
| 문서 저장 | S3 |
| 임베딩/청킹 | Bedrock KB (자동 chunking/embedding) |
| 벡터 스토어 | OpenSearch Serverless |
| 응답 생성 | Bedrock Claude |

- **장점**: 운영 부담 최소, 빠른 구축 (2-3주)
- **단점**: 커스터마이징 제한, 지연시간 제어 어려움
- **TTFT**: 1-3초 (제어 불가) — **속도 요구사항 미달 위험**

### Option B: Custom RAG on EKS

| 구성요소 | AWS 서비스 |
|---------|-----------|
| 문서 저장 | S3 |
| 임베딩 | SageMaker 또는 Bedrock Embeddings |
| 벡터 스토어 | OpenSearch / pgvector on Aurora |
| 시맨틱 캐싱 | ElastiCache (Redis) |
| 오케스트레이션 | EKS 커스텀 서비스 |
| 응답 생성 | Bedrock Claude |

- **장점**: 지연시간 최적화, 완전한 제어
- **단점**: 운영 복잡도 높음, 구축 6-8주
- **TTFT**: ~500ms (최적화 가능) — 구축 기간 과다

### Option C: 하이브리드 (Recommended)

| 구성요소 | AWS 서비스 | 접근 방식 |
|---------|-----------|----------|
| 문서 수집/임베딩 | Bedrock KB | 관리형 |
| 벡터 스토어 | OpenSearch Serverless | 관리형 |
| 실시간 오케스트레이션 | EKS 커스텀 서비스 | 커스텀 |
| 시맨틱 캐싱 | ElastiCache | 커스텀 |
| 응답 생성 | Bedrock Claude (스트리밍) | 관리형 |

- **장점**: 핵심은 관리형으로 운영 부담 최소화, 성능 크리티컬 부분만 커스텀
- **구축 기간**: 3-4주
- **TTFT**: ~500ms + 구축 3-4주 — 속도와 구축 속도 균형
- **선택 이유**: 초기 구축 속도와 성능 최적화 사이의 최적 균형점. 속도 요구사항(2초 이내)을 충족하면서 관리형 서비스로 운영 부담 최소화

> **참고: Claude Code CLI는 프로덕션 배포 부적합**
> Claude Code는 개발자용 CLI 도구이며 서버 배포를 위한 API가 아님. 프로덕션 콜센터 시스템에서는 **Bedrock API를 직접 호출** (`InvokeModelWithResponseStream`)하여 스트리밍 응답을 생성하는 것이 올바른 접근. EKS 서비스에서 boto3/AWS SDK로 Bedrock API를 호출하는 구조.

## 6. 핵심 컴포넌트 상세

### A. 문서 수집 파이프라인 (Offline)

```
[S3 버킷] ← Word, Wiki export, PDF 업로드
    ↓
[EventBridge] → [Lambda / Step Functions] → 전처리 (텍스트 추출, 정제)
    ↓
[Bedrock KB] → 자동 chunking / embedding
    ↓
[OpenSearch Serverless] → 인덱싱
```

- S3 버킷에 Word, Wiki export, PDF 업로드
- EventBridge → Lambda/Step Functions → 전처리 (텍스트 추출, 정제)
- Bedrock KB 자동 chunking/embedding
- OpenSearch Serverless 인덱싱
- 주기적 Wiki 크롤러 (Lambda + EventBridge Scheduler)

### B. 실시간 처리 레이어 (Online, EKS)

- **STT 텍스트 → WebSocket Gateway** 수신
- **대화 버퍼**: 슬라이딩 윈도우 (~30초) + 전체 통화 컨텍스트 유지
- **대화 요약**: 긴 통화의 컨텍스트 압축 (Bedrock Haiku로 저비용 처리)
- **선제적 검색**: 키워드 기반 문서 프리페치 — 고객이 질문하기 전에 관련 문서를 미리 조회
- **비동기 파이프라인**: 의도 감지와 선제적 검색을 병렬 실행 (직렬 X)
- **검색 결과 프리로드**: 대화 키워드 감지 즉시 top-K 문서를 메모리에 프리페치
- **컨텍스트 윈도우 최적화**: 요약본 + 최근 3턴만 전송 → input token 최소화 → Bedrock 응답 속도 향상

#### B-1. 3단계 질문 감지 파이프라인

```
STT 텍스트 스트림
    ↓
[L1: 룰 기반 감지] ← ~5ms, 무비용
  - 한국어 종결어미 패턴 (의문형)
  - 의문사 키워드 매칭
    ↓ (L1 미감지 시)
[L2: 경량 ML 분류기] ← ~20ms, 로컬
  - ONNX 모델 (문장 분류)
  - 콜센터 도메인 fine-tuned
    ↓ (L2 불확실 시)
[L3: Bedrock Haiku] ← ~200ms, 저비용
  - 복합 의도 분석
  - 컨텍스트 의존적 판단
```

**목표**: L1에서 70-80% 감지, L2에서 15-20%, L3는 5-10% 폴백

#### B-2. L1 룰 기반 — 한국어 질문 패턴 상세

| 카테고리 | 패턴 예시 | 정규식 접근 |
|----------|----------|------------|
| **의문형 종결어미** | ~인가요?, ~인지요?, ~나요?, ~까요? | `(인가요\|인지요\|나요\|까요\|ㄹ까요\|을까요)\s*[?]?$` |
| **의문사 시작** | 어떻게, 왜, 언제, 어디서, 얼마나 | `^(어떻게\|왜\|언제\|어디\|얼마\|몇\|무엇\|뭐)\s` |
| **요청형** | ~해주세요, ~알려주세요, ~설명해 | `(해주세요\|알려주세요\|설명해\|말씀해)\s*$` |
| **확인형** | ~맞나요?, ~되나요?, ~가능한가요? | `(맞나요\|되나요\|가능한가요\|있나요)\s*[?]?$` |
| **불만/문제 신호** | 안 돼요, 문제가, 오류가 | `(안\s?돼\|문제가\|오류가\|에러가\|고장)` |

**STT 특화 보정**:
- 물음표 없는 의문문 처리 (STT는 `?` 미출력하는 경우 많음)
- 억양/포즈 정보 활용 불가 → 종결어미에 의존도 높음
- "~거든요" (설명) vs "~거든요?" (질문) 구분이 어려움 → L2로 에스컬레이션

#### B-3. 오감지(False Positive) 처리

- **수사적 질문 필터**: "그렇죠?" → 동의 구하기, RAG 트리거 불필요
- **자기 대답 패턴**: 질문 직후 본인이 이어서 답하는 경우 → 500ms 대기 후 트리거
- **반복 질문 디바운싱**: 동일 의도 질문 3초 내 재발화 시 무시
- **상담원 발화 구분**: 화자 분리(speaker diarization)로 고객 발화만 감지 대상

#### B-4. 질문 유형 분류 (RAG 쿼리 최적화용)

| 유형 | 설명 | RAG 전략 |
|------|------|----------|
| **상품/서비스 문의** | "이 상품 환불 가능한가요?" | 상품 KB 우선 검색 |
| **절차/프로세스 문의** | "신청 방법이 어떻게 되나요?" | 프로세스 문서 우선 |
| **기술 지원** | "앱이 안 열려요" | 트러블슈팅 가이드 우선 |
| **정책/규정 문의** | "위약금이 얼마인가요?" | 정책 문서 + 최신 약관 |
| **불만/클레임** | "왜 이렇게 오래 걸려요?" | 에스컬레이션 프로토콜 + 위로 스크립트 |

유형별로 검색 필터를 달리하면 검색 정확도 향상 + 불필요한 문서 제거 → 속도 개선

### C. RAG + 응답 생성

1. **시맨틱 캐시 우선 경로**: 캐시 히트 시 Bedrock 호출 스킵 → 응답시간 ~100ms
2. **Query Rewriting**: 대화 컨텍스트 → 검색 최적화 쿼리로 변환
3. **병렬 검색**: BM25 + kNN 동시 실행 후 merge (직렬 대비 40% 단축)
4. **Reranking**: 검색 결과 관련도 기반 재정렬
5. **Bedrock 커넥션 풀**: 사전 수립된 연결 재사용 (콜드스타트 제거)
6. **Bedrock Claude 스트리밍 응답**: TTFT 목표 **500ms**, 전체 답변 **2초**
7. **출처 문서 링크 포함**: 상담원이 필요 시 상세 확인 가능

### E. 답변 지연 시 해결 전략

#### E-1. 프로그레시브 답변 표시 (상담원 UI)

```
[즉시] 질문 감지 표시: "고객 질문 감지: 환불 절차 문의"
   ↓ ~100ms
[캐시 히트 시] 캐시된 답변 즉시 표시
   ↓ ~300-500ms
[검색 완료] 관련 문서 스니펫 먼저 표시 (Bedrock 응답 대기 없이)
   ↓ ~500ms~
[스트리밍] Bedrock 응답 토큰 단위로 점진 표시
```

핵심: **Bedrock 응답을 기다리지 않고 검색 결과만으로도 상담원에게 즉시 가치 제공**

#### E-2. 다단계 폴백 전략

| 상황 | 지연 | 폴백 액션 |
|------|------|----------|
| 캐시 히트 | ~100ms | 캐시 답변 표시, Bedrock 스킵 |
| 캐시 미스 + 검색 빠름 | ~500ms | 검색 결과 스니펫 먼저 표시 → Bedrock 스트리밍 이어서 |
| Bedrock 응답 느림 (>2초) | >2초 | "답변 생성 중..." + 검색 문서 링크 우선 표시 |
| Bedrock 타임아웃 (>5초) | >5초 | 검색 결과만 표시 + "수동 검색" 버튼 |
| 전체 파이프라인 장애 | - | 기존 검색 시스템 폴백 (키워드 검색) |

#### E-3. 3계층 캐시 구조

```
[L1: 인메모리 캐시] ← 현재 통화 컨텍스트 (동일 통화 내 반복 질문)
  - TTL: 통화 종료까지
  - 히트율: ~10-15%
    ↓
[L2: 시맨틱 캐시 (Redis)] ← 임베딩 유사도 기반 (cosine > 0.92)
  - TTL: 24시간 (업무 문서 변경 주기에 맞춤)
  - 히트율: 안정기 30-40%
    ↓
[L3: 인기 질문 캐시 (Redis)] ← FAQ 기반 사전 생성 답변
  - TTL: 7일 (수동 갱신)
  - 히트율: ~15-20%
```

**총 캐시 히트율 목표**: 안정기 **50-60%** → Bedrock 호출 절반으로 감소

**캐시 프리히팅**:
- 매일 업무 시작 전 자동 실행 (EventBridge Scheduler)
- 전일 top-50 질문으로 캐시 워밍
- 신규 문서 업로드 시 관련 캐시 무효화

### D. 상담원 UI (React SPA on CloudFront)

- **실시간 추천 답변 패널**: WebSocket 스트리밍으로 답변이 점진적으로 표시
- **대화 실시간 트랜스크립트 뷰**: 고객/상담원 발화 구분 표시
- **관련 문서 사이드바**: 출처 문서 목록 + 핵심 구절 하이라이트
- **피드백 버튼**: 유용/부적절 평가 → 품질 개선 루프에 반영

## 8. 속도 아키텍처 + 데이터 플라이휠

### 속도 아키텍처

| 기법 | 설명 | 예상 효과 |
|------|------|----------|
| 시맨틱 캐싱 | ElastiCache에 임베딩 유사도 기반 캐시 | 반복 질문 90%+ 응답시간 단축 |
| 선제적 검색 | 대화 키워드 기반 문서 프리페치 | 질문 감지 시 이미 문서 준비됨 |
| 스트리밍 | Bedrock streaming API → 첫 토큰 ~500ms | 체감 응답시간 대폭 단축 |
| 연결 풀링 | Bedrock API 커넥션 재사용 | 연결 수립 오버헤드 제거 |
| 리전 내 배치 | 모든 서비스 ap-northeast-2 | 네트워크 지연 최소화 |

### 속도 SLO 및 운영

- **지연시간 버짓 관리**: 각 컴포넌트별 SLO 설정 + CloudWatch 대시보드 모니터링
- **P99 목표**: 질문→첫 토큰 1초, 질문→전체 답변 3초 (P50: 2초)
- **성능 저하 시 폴백**: 캐시 미스 + Bedrock 지연 시 → 검색 결과만 우선 표시 (답변 생성 대기 없이)
- **Warm-up 전략**: 업무 시작 전 자주 사용되는 쿼리로 캐시 프리히팅

### 데이터 플라이휠 (지속 개선)

```
상담원 피드백 → 답변 품질 평가 → 프롬프트 튜닝
       ↓
미응답 질문 → 지식 갭 리포트 → 문서 보강
       ↓
통화 로그 분석 → FAQ 자동 생성
       ↓
A/B 테스트 → 프롬프트/검색 파라미터 최적화
       ↓
(반복)
```

- **상담원 피드백**: 유용/부적절 평가 데이터 → 프롬프트 튜닝에 활용
- **미응답 질문**: RAG가 답변하지 못한 질문 → 지식 갭 리포트 → 문서 보강 우선순위
- **통화 로그 분석**: 빈출 질문 패턴 → FAQ 자동 생성 → 캐시 적중률 향상
- **A/B 테스트**: 프롬프트 버전, 검색 파라미터 등 지속 최적화

## 10. 예상 질문과 답변 (FAQ)

프레젠테이션/검토 시 예상되는 질문에 대한 답변 준비:

| 예상 질문 | 답변 요지 |
|----------|----------|
| **질문 감지 정확도는?** | L1 룰 기반 70-80% + L2 ML 15-20% + L3 LLM 5-10%. 총 정확도 목표 95%+. 오감지(false positive)보다 미감지(false negative)가 더 심각 → recall 우선 튜닝 |
| **답변이 늦으면?** | 프로그레시브 표시 — 검색 결과 먼저(~500ms), Bedrock 스트리밍 이어서. 최악의 경우에도 관련 문서 링크는 1초 내 표시 |
| **답변 정확도는?** | RAG + Reranking + 출처 문서 명시. 환각(hallucination) 방지: 검색 결과 기반으로만 답변 생성하도록 프롬프트 제약. PoC에서 70%+ 목표, 프로덕션 85%+ |
| **캐싱은 어떻게?** | 3계층 (인메모리/시맨틱/FAQ), 안정기 50-60% 히트율 → Bedrock 비용 및 지연시간 동시 절감 |
| **150명 동시 처리 가능?** | EKS 오토스케일링 + Bedrock on-demand. 병목은 Bedrock 동시 요청 수 → 캐시로 실 호출 50% 감소. Provisioned Throughput 옵션도 가능 |
| **기존 CTI와 연동 어려움은?** | 프로토콜 어댑터 패턴 — CTI vendor별 플러그인. STT 텍스트만 WebSocket으로 수신하면 되므로 연동 범위 최소화 |
| **보안은?** | VPC 내 배포, Bedrock은 데이터 학습 미사용, 고객 개인정보 마스킹 전처리, DynamoDB 암호화 |
| **Claude Code로 만들면 안 돼요?** | Claude Code는 개발자 CLI 도구. 프로덕션에서는 Bedrock API (`InvokeModelWithResponseStream`)를 EKS 서비스에서 직접 호출하는 것이 올바른 접근 |
