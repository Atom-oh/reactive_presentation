### 멀티 클러스터 운영방안

- 현재는 management 목적의 단일 클러스터만 최초 구성 이후 비주요 워크로드(MCP Gateway, 부하 테스트 환경)에만 운영중이나, ECS로 운영되고 있는 전체 서비스를 이관하면서 멀티 클러스터 구조로 확장을 계획하고 있음
- 멀티 클러스터 운영시 현재 AWS 어카운트와 네트워크, 서비스가 분리된 구조에 따라 어떤 기준으로 구분하는것이 좋을지 문의
- 문의를 통해 정리된 구조를 효율적으로 점진적 적용을 위한 장기계획

### EKS 및 Kubernetes 리소스를 Immutable infra로 관리하기 위한 IaC 전략

- Terraform과 ArgoCD를 혼합한 현재 구성방안과 원칙(확대)
    
    ```mermaid
    flowchart TB
        subgraph IaC["Infrastructure as Code"]
            direction TB
    
            subgraph ArgoCD["ArgoCD (GitOps)"]
                direction LR
                subgraph Apps["애플리케이션"]
                    K8sResources["Deployments, Services"]
                    CRDs["Custom Resources"]
                end
                subgraph Controllers["클러스터 컨트롤러"]
                    ALB["AWS Load Balancer Controller"]
                    ExtDNS["External DNS"]
                    ExtSecrets["External Secrets"]
                    CertManager["Cert Manager"]
                end
            end
    
            subgraph Terraform["Terraform"]
                direction LR
                subgraph Infra["기반 인프라"]
                    EKSCluster["EKS 클러스터"]
                    VPC["VPC / Subnet"]
                end
                subgraph AWSResources["AWS 리소스"]
                    RDS["RDS"]
                    SG["Security Groups"]
                    IAM["IAM Roles"]
                end
            end
        end
    
        Git["Git Repository"] --> ArgoCD
        Git --> Terraform
        Apps --> EKS["EKS Cluster"]
        Terraform --> AWS["AWS"]
    
        Controllers -->|"AWS API 호출"| AWS
    ```
    
    1. **EKS 내 리소스는 ArgoCD Application으로 관리**
        - 클러스터 Controller와 Application 모두 ArgoCD로 배포한다.
        - 예외: VPC CNI, CoreDNS 등과 같이 직접 Controller로 구성하는것보다 EKS Add-on으로 구성하는것이 적절한 도구는 Terraform으로 관리한다.
    2. **ArgoCD Application 전용 AWS 리소스는 매니페스트와 함께 관리**
        - Application 전용 AWS 리소스는 기본적으로 ArgoCD를 통해 구성한다.
        - Application에 종속되지만 ArgoCD로 구성이 불가능한 리소스는 `terraform/applications/<app>/` 디렉토리에 배치하고 Terraform으로 관리한다.
        - e.g. Security Group, IAM Role
    3. **공용 AWS 리소스는 Terraform으로 관리**
        - 여러 애플리케이션이 공유하거나 애플리케이션과 독립적인 생명주기를 가진 리소스는 `terraform/<resource>/` 디렉토리에 배치하고 Terraform으로 관리한다.
        - e.g. EKS 클러스터, 공용 RDS
- ACK를 활용한 Kubernetes 네이티브 AWS 리소스 관리 방안
- Self-managed가 아닌 [EKS Capabilities](https://docs.aws.amazon.com/eks/latest/userguide/capabilities.html)의 ArgoCD와 ACK 선택을 위한 의사결정 요소

### Node별 Pod 할당 제한이슈

- SGP가 활성화된 pod의 경우, Branch ENI로만 할당이 가능한 한계로 인해 node별 할당 가능한 pod가 부족함(현재는 일부 pod에만 SGP를 적용중)
    - SGP를 활용해야만 pod별 sg 설정이 가능한것인지?
- SGP를 사용하지 않는 케이스에서도 인스턴스 타입에 따라 ENI당 최대 IP 할당이 제한되는 문제
    - 리소스가 작은 pod가 많이 밀집되는 node의 경우 시스템 리소스는 여유가 있으나 IP 할당 제한으로 pod 할당 불가
- 사전에 알아본 해결방안
    - Prefix Delegation 활성화를 통해 secondary eni에 16배의 pod 할당 방식
    - spg 설정이 필요한 pod를 fargate로 전환
    - 인스턴스 타입 스케일업

### 서브넷 IP 고갈 문제 사전 대응 필요여부

- VPC Secondary CIDR + Custom Networking 방식을 통해 VPC에서 할당 가능한 IP의 범위 자체를 늘릴 수 있음
- 클러스터 설계 초기 시점부터 고려를 해야할지? 운영하면서 문제발생시 추후 적용이 가능할지?
