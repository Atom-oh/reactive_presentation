Taint/Toleration으로 워크로드 격리

하이브리드 노드에 특정 Pod만 스케줄링

YAML # 노드 Taint 설정 (nodeadm flags) --register-with-taints=eks.amazonaws.com/compute-type=hybrid:NoSchedule # Pod Toleration tolerations: - key: eks.amazonaws.com/compute-type operator: Equal value: hybrid effect: NoSchedule # GPU 노드 추가 Taint --register-with-taints=gpu=true:NoSchedule
핵심: NoSchedule Taint가 있는 노드에는 해당 Toleration이 있는 Pod만 스케줄링됨

GPU 온프레미스 + CPU 클라우드

ML Training은 온프렘 GPU, Inference는 클라우드 CPU

패턴: GPU 투자는 보존하고, 추론/서빙은 클라우드 탄력성 활용 - 모델 아티팩트는 S3 경유

Cloud Bursting with Karpenter

온프레미스 용량 초과 시 클라우드로 자동 확장

Scale-up: 온프렘 가득 차면 Karpenter가 클라우드 노드 프로비저닝
Scale-down: pod-deletion-cost로 클라우드 Pod 먼저 제거

pod-deletion-cost.yaml # 온프렘 Pod - 높은 deletion cost apiVersion: v1 kind: Pod metadata: annotations: controller.kubernetes.io/pod-deletion-cost: "1000" --- # 클라우드 Pod - 낮은 deletion cost apiVersion: v1 kind: Pod metadata: annotations: controller.kubernetes.io/pod-deletion-cost: "-100"

데이터 주권 보장

민감 데이터는 온프레미스에 유지, Control Plane만 AWS에서 관리 → 금융 규제 충족

감사 추적

IAM Roles Anywhere의 x509Subject/CN으로 노드별 감사 추적 가능 → 인증서 기반 식별

네트워크 격리

VPN/Direct Connect + Private Endpoint로 퍼블릭 인터넷 경유 없음 → 보안 경계 유지

컴플라이언스 자동화

AWS Config, CloudTrail로 Control Plane 관련 컴플라이언스 자동 모니터링

온프렘 단독 vs EKS Hybrid + AWS 서비스

Before: 온프렘 단독

  • LLM 모델 자체 호스팅 필요 (GPU 인프라 구축)
  • 모델 업데이트/파인튜닝 직접 수행
  • 벡터 DB, 검색 엔진 별도 구축
  • Agent 파이프라인 처음부터 개발
  • 보안/접근 제어 직접 구현
수개월 소요

After: EKS Hybrid + AWS

  • Bedrock API로 최신 모델 즉시 활용 (Claude, Nova)
  • 모델 관리/업데이트는 AWS가 담당
  • Kendra/OpenSearch로 RAG 즉시 구성
  • Bedrock Agent로 파이프라인 자동 구성
  • IAM 기반 세분화된 접근 제어
수주 내 PoC 가능
ROI: 인프라 구축 시간을 서비스 개발에 집중 - 비즈니스 가치 창출 가속화
