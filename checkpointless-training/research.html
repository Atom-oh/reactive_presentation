SageMaker HyperPod Checkpointless Training은 2025년 12월 re:Invent에서 발표된 신기능으로, 대규모 분산 학습에서 전통적인 체크포인트 기반 복구를 peer-to-peer 인메모리 상태 복제로 대체합니다.

핵심 혁신: 장애 발생 시 디스크에서 체크포인트를 로딩하는 대신, 건강한 피어 노드에서 EFA 고속 네트워크를 통해 모델/옵티마이저 상태를 직접 복구합니다. Amazon Nova 모델이 수만 개의 가속기에서 이 기술로 학습되었습니다.

2. 분산 학습과 Checkpointing 기초

2.1 분산 학습이란?

신경망 학습은 반복적인 과정입니다. 각 학습 스텝은 다음 단계로 구성됩니다:

  1. Forward Pass: 입력 데이터가 모델 레이어를 통과하며 예측값 생성
  2. Loss Calculation: 예측값과 정답의 차이(손실) 측정
  3. Backward Pass: 손실을 역방향으로 전파하여 각 파라미터의 그래디언트 계산
  4. Gradient Synchronization: 분산 환경에서 모든 GPU의 그래디언트를 동기화
  5. Optimizer Step: 그래디언트를 기반으로 모델 파라미터 업데이트
SA를 위한 비유

Data Parallelism = ALB 뒤 동일한 EC2 인스턴스들이 각각 다른 요청 처리
Tensor Parallelism = 하나의 DB 쿼리를 여러 샤드에서 분할 처리
Pipeline Parallelism = Step Functions에서 각 Lambda가 한 단계 담당
FSDP/ZeRO = S3 multipart upload (분할 저장, 필요 시 재조립)

2.2 병렬화 유형

유형분할 대상메모리 이점통신 오버헤드사용 시점
Data Parallelism (DDP)입력 배치없음 (모델 전체 복제)Gradient All-Reduce모델이 단일 GPU에 적합
Tensor Parallelism (TP)개별 레이어 연산분할 비례높음 (매 연산마다)단일 레이어가 GPU에 안 들어감
Pipeline Parallelism (PP)모델 레이어를 스테이지로스테이지 비례중간 (스테이지 간)모델이 크지만 레이어는 적합
ZeRO / FSDP옵티마이저, 그래디언트, 파라미터최대 N배 (GPU 수)추가 All-Gather/Reduce-Scatter대형 모델, 메모리 제약

2.3 Checkpointing의 역할

Checkpointing은 학습 상태를 영구 스토리지에 주기적으로 저장하는 내결함성 메커니즘입니다. 체크포인트에 저장되는 항목:

구성요소크기 (파라미터당)설명
Model Parameters2 bytes (FP16)실제 신경망 가중치
Optimizer State (Adam)8-12 bytes1st moment (momentum), 2nd moment (variance), FP32 master weights
Gradient Buffers2-4 bytes옵티마이저 스텝 전 누적된 그래디언트
기타 메타데이터소량LR 스케줄러, epoch, batch 번호, 데이터로더 상태
핵심 수치

70B 파라미터 모델의 체크포인트 크기:
파라미터만 (FP16): 70B × 2 bytes = 140 GB
Adam 옵티마이저 포함: 70B × ~7.4 bytes = 521 GB
전체 상태 (활성화 제외): ~1 TB

3.3 복구 프로세스 - 6단계 순차 블로킹

전통적 체크포인트 기반 복구는 모든 단계가 순차적이고 블로킹입니다:

  1. Stage 1: Job 종료 및 재시작 - 오케스트레이터가 모든 노드의 모든 프로세스를 종료
  2. Stage 2: 프로세스/네트워크 초기화 - 학습 스크립트 재실행, NCCL 초기화 → 대형 클러스터에서 수십 분
  3. Stage 3: 체크포인트 로딩 - S3/FSx에서 수백 GB~TB 로딩 → 수십 분~수 시간
  4. Stage 4: 데이터로더 초기화 - 데이터 프리페치, 버퍼 초기화 → 수 분
  5. Stage 5: First Step Overhead (FSO) - CUDA 컨텍스트, 메모리 할당, 그래프 컴파일
  6. Stage 6: Lost Steps Overhead (LSO) - 마지막 체크포인트 이후 모든 스텝 재계산

4. SageMaker HyperPod Checkpointless Training

출시일: 2025년 12월 3일 (re:Invent 2025)
추가 비용: 없음 (HyperPod 사용 비용에 포함)
요구사항: Amazon EKS 오케스트레이션, Training Operator v1.2.0+

4.1 패러다임 전환

기존: 디스크에 저장 → 디스크에서 복원
Checkpointless: 인메모리 P2P 상태 복제 → 피어에서 직접 복원

4.2 Component 1: Rootless 초기화 (TCPStore-less)

해결하는 단계: Stage 2 (프로세스/네트워크 초기화)

문제

  • 기존 torch.distributed는 TCPStore를 rendezvous 포인트로 사용
  • 모든 rank가 rank 0 (루트 서버)에 동시 접속 → 네트워크 병목
  • 대형 클러스터 초기화: 수십 분

해결

  • 대칭 주소 패턴: 각 rank가 글로벌 그룹 카운터를 사용해 피어 연결 정보를 독립적으로 계산
  • 미리 결정된 포트 할당으로 직접 연결
  • 단일 장애점(SPOF) 제거
  • Checkpointless 초기화 시간: 수 초
# Kubernetes 환경변수 설정
export HPCT_USE_ROOTLESS=1
export TORCH_SKIP_TCPSTORE=1
sysctl -w net.ipv4.ip_local_port_range="20000 65535"

4.3 Component 2: Memory-Mapped Data Loading (MMAP)

해결하는 단계: Stage 4 (데이터로더 초기화)

문제

  • 프로세스 재시작 시 디스크에서 배치 재로딩
  • 데이터로더 상태 재구축 필요
  • 중복 샘플/누락 위험

해결

  • 학습 데이터를 /dev/shm (공유 메모리)에 매핑
  • 프로세스가 실패해도 메모리 영역은 유지
  • 복구된 노드가 기존 캐시에 바로 재접속
  • 메모리 사용량: 노드당 8개 → 1개 복사본 (8-GPU 노드 기준)
from hyperpod_checkpointless_training.dataloader.mmap_data_module import MMAPDataModule
from hyperpod_checkpointless_training.dataloader.config import CacheResumeMMAPConfig

data_module = MMAPDataModule(
    data_module=MY_DATA_MODULE(...),
    mmap_config=CacheResumeMMAPConfig(
        cache_dir="/dev/shm/pdl_cache",
        prefetch_length=10,
        lookback_length=2,
    ),
)

4.4 Component 3: In-Process Recovery

해결하는 단계: Stage 1, 2, 5 (Job 종료, 프로세스 초기화, FSO)

문제

  • 단일 GPU 오류가 클러스터 전체 셧다운을 트리거
  • 모든 프로세스가 종료 후 재시작 필요

해결

  • 장애를 프로세스 수준에서 격리
  • 실패한 프로세스가 수 초 내에 학습 루프에 재합류
  • 건강한 프로세스는 중단 없이 계속 실행
  • CUDA 컨텍스트, 컴파일러 캐시, GPU 상태 보존
  • 복구 불가능한 오류: pre-warmed hot spare로 자동 교체

장애 유형별 복구 계층

장애 유형원인복구 방식메커니즘
In-Process Failure코드 오류, 예외IPRRCB(Re-executable Code Block) 재실행
Process RestartCUDA 컨텍스트 손상PLRTraining Operator가 프로세스만 재시작
Node Replacement영구 하드웨어 장애JLR노드 교체 후 Job 재시작
from hyperpod_checkpointless_training.inprocess.wrap import HPWrapper
from hyperpod_checkpointless_training.inprocess.health_check import CudaHealthCheck
from hyperpod_checkpointless_training.inprocess.train_utils import wait_rank, HPAgentK8sAPIFactory

wait_rank()  # Training Operator에서 rank 정보 대기

@HPWrapper(
    health_check=CudaHealthCheck(),
    hp_api_factory=HPAgentK8sAPIFactory(),
    abort_timeout=60.0,
    checkpoint_manager=CheckpointManager(enable_offload=False),
)
def run_main(cfg, caller=None):
    trainer = Trainer(
        strategy=CheckpointlessMegatronStrategy(
            num_distributed_optimizer_instances=2  # 최소 2 (중복성)
        ),
    )
    trainer.wrapper = caller

4.5 Component 4: Peer-to-Peer State Replication

해결하는 단계: Stage 3, 6 (체크포인트 로딩, Lost Steps)

문제

  • S3/FSx에서 수백 GB 모델/옵티마이저 상태 로딩: 수십 분

해결

  • N+1 또는 M-copy redundancy 방식으로 모델 상태를 노드 그룹 간 복제
  • GPU VRAM에서 피어 노드의 CPU DRAM으로 미러링
  • num_distributed_optimizer_instances ≥ 2 (hybrid sharding)
  • 장애 시 건강한 피어에서 EFA (최대 3,200 Gbps)를 통해 상태 전송
  • 디스크 I/O 완전 제거: 복구에 디스크 접근 불필요
  • 복구 시간: 수 초
통신 패턴 변경

기존: reduce-scatter + all-gather (전체 DP 그룹)
변경 후: reduce-scatter (복제 그룹 내) + all-reduce (복제 그룹 간) + all-gather (복제 그룹 내)
결과: 동일한 총 통신량, 더 작은 collective 그룹 = 더 나은 레이턴시

4.6 Component 5: HyperPod Training Operator

Kubernetes 확장으로 모든 컴포넌트를 오케스트레이션합니다:

  • 학습 프로세스 건강 상태의 글로벌 뷰 유지
  • 장애 감지 → 복구 결정 → 클러스터 동기화 조율
  • 지능형 복구 에스컬레이션: IPR → PLR → JLR 순서로 시도
  • HyperPod 건강 모니터링 에이전트와 통합
  • Hanging job 감지 및 커스텀 복구 정책

PyTorch Elastic (torchrun)

동적 워커 스케일링과 내결함성을 지원하지만, 장애를 Job 수준 이벤트로 처리합니다. 모든 프로세스가 종료 후 재시작되며, 여전히 체크포인트 기반 복구에 의존합니다.

NVIDIA Resiliency Extension (NVRx)

PyTorch용 내결함성 프레임워크로 Hung Rank Detection, In-Job Restart, Straggler Detection, 비동기 체크포인팅을 제공합니다. PyTorch Lightning 및 NeMo와 통합되지만, 상태 복제는 수동 통합이 필요합니다. 아직 실험적 단계입니다.

DeepSpeed Checkpointing

파이프라인 스테이지 병렬 쓰기, ZeRO Stage 3 최적화, 노드 로컬 스토리지 옵션을 제공합니다. 하지만 근본적으로 디스크 기반이며, S3/디스크 I/O 병목을 제거하지 못합니다.

Google Optical Circuit Switches (OCS)

장애 주변의 네트워크 토폴로지를 하드웨어 수준에서 재구성합니다. AWS는 정적/hot-spare 패브릭 위에서 소프트웨어 정의 인메모리 복제에 집중합니다.

7.2 예상 Q&A

Q1: Checkpointless면 체크포인트를 아예 안 하나요?

A: "장애 복구를 위한 체크포인트는 필요 없지만, 학습 완료 후 모델 저장이나 중간 결과 보존을 위한 체크포인트는 여전히 할 수 있습니다. 핵심은 장애 복구 경로에서 디스크 I/O를 제거한 것입니다."

Q2: 메모리 오버헤드는 얼마나 되나요?

A: "num_distributed_optimizer_instances=2로 설정하면, 고정밀 master weights와 옵티마이저 상태가 2벌 유지됩니다. 대략 DCP 체크포인트 크기 1개분의 추가 메모리입니다. 저정밀 weights, gradients, activation은 기존과 동일하게 sharding됩니다."

Q3: 어떤 프레임워크를 지원하나요?

A: "NeMo, PyTorch, PyTorch Lightning을 지원합니다. 사전 구성된 레시피가 Llama 3 70B, GPT OSS 120B에 제공되며, 커스텀 모델도 @HPWrapper 데코레이터와 CheckpointlessMegatronStrategy로 통합 가능합니다."

Q4: Slurm도 지원하나요?

A: "현재 Checkpointless Training은 Amazon EKS 오케스트레이션 전용입니다. HyperPod Training Operator v1.2.0+ 이 필요합니다. Slurm 기반 HyperPod에서는 Managed Tiered Checkpointing을 사용할 수 있습니다."

Q5: 기존 학습 코드를 얼마나 수정해야 하나요?

A: "HyperPod Recipes를 사용하면 Llama, GPT-OSS 등 인기 모델은 코드 변경 제로입니다. 커스텀 모델의 경우 4-tier 접근법으로 점진적 통합이 가능합니다: Tier 1은 환경변수만, Tier 4까지 가면 full checkpointless입니다."

Q6: Training loss나 수렴에 영향이 있나요?

A: "없습니다. AWS에서 기존 방식과 Checkpointless 방식의 training loss를 비교했고, 모든 스텝에서 bit-wise 일치를 확인했습니다. RNG 상태까지 보존하므로 결정론적 복구가 가능합니다."

Q7: 비용은 얼마인가요?

A: "추가 비용 없이 HyperPod에서 사용 가능합니다. SageMaker HyperPod가 제공되는 모든 리전에서 사용할 수 있습니다."
