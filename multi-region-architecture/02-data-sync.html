MSK Replicator: Cross-Region

[us-east-1 MSK] ─── MSK Replicator ──→ [us-west-2 MSK]
(IAM Auth, async)
  • 복제 대상: 모든 토픽 (regex: .*)
  • Consumer Offset 동기화 가능
  • Compression: GZIP
  • Latency: 수백 ms ~ 수 초

장점

  • DR 시 Secondary에서 즉시 consume 가능
  • Consumer offset 동기화 → 이벤트 유실 최소화
  • 토픽 구성 자동 동기화

비용

  • 데이터 전송: $0.02/GB (리전 간)
  • Replicator 시간당 요금 추가
[us-east-1 MSK] ─── (no replication) ──→ [us-west-2 MSK]
  • Secondary MSK: 독립 클러스터 (빈 토픽)
  • DR 시 Producer가 Secondary로 전환

장점

  • 크로스리전 데이터 전송 비용 없음
  • 구성 단순

단점

  • DR 시 이벤트 유실 불가피 (in-flight)
  • Consumer offset 재설정 필요
  • Failover 시간 증가

Key Takeaways

1. Polyglot Persistence

각 데이터 스토어는 워크로드 특성에 맞게 선택. Aurora(ACID), DocumentDB(문서), ElastiCache(캐시), MSK(이벤트)

2. Write-Primary / Read-Local

모든 쓰기는 Primary 리전으로 라우팅. 읽기는 양 리전에서 서비스. Aurora Write Forwarding 활용.

3. Async Replication Trade-offs

모든 크로스리전 복제는 비동기. RPO는 1초(Aurora)~15분(S3). 강한 일관성 필요시 Primary에서 처리.

4. Idempotency is King

Failover, 복제 지연, 네트워크 파티션 — 모든 상황에서 안전하려면 서비스의 idempotent 설계가 필수.

Block 2 Quiz

Q1. Aurora Global Write Forwarding에서 3개 statement 트랜잭션의 추가 레이턴시는?

60-80ms (1 RTT)
120-160ms (2 RTT)
180-240ms (3 RTT)
240-320ms (4 RTT)
각 statement마다 RTT가 추가됩니다. 3개 statement = 3 RTT ≈ 180-240ms.

Q2. DocumentDB Global Cluster에서 Secondary 리전의 쓰기 처리 방식은?

Write Forwarding으로 자동 포워딩
애플리케이션이 직접 Primary 엔드포인트로 요청
Secondary에서 직접 쓰기 후 동기화
Active-Active 양방향 쓰기
DocumentDB는 Write Forwarding 미지원. 쓰기는 애플리케이션이 Primary로 직접 라우팅해야 합니다.

Q3. 재고 카운터(stock)에 TTL을 설정하지 않는 이유는?

비용 절감
원자적 DECR로 실시간 재고 관리
Global Datastore가 TTL 미지원
복제 시 TTL 동기화 불가
재고는 DECR로 원자적 감소. TTL 만료 시 재고 데이터 유실 → 초과 판매 위험.

Q4. 멀티리전 데이터 아키텍처에서 가장 중요한 설계 원칙은?

강한 일관성 보장
최소 레이턴시
Idempotent 설계
동기식 복제
